1994 HARDWARE · 2026 KERNEL · ONE STATIC PAGE

Old machines make unusually good software tests.

This started as a way to run current mainline Linux on an Acorn RISC PC without asking anyone to find a working thirty-two-year-old computer. It became a hardware-assisted archaeology project: model the machine from its data sheets, run independent operating systems, and treat every disagreement as evidence instead of tuning the emulator until one driver is happy.

The result is the machine on the previous page: a real ARMv4 Linux kernel running inside QEMU, compiled to WebAssembly, with its own VIDC20 screen, keyboard, mouse, IDE disk, floppy controller and a separate serial VT220. Nothing runs on a backend server.

METHOD

The data sheet is the specification

Independent oracles

The VIDC20 and SA-110 manuals define the model. Mainline Linux is the fast feedback loop; NetBSD/acorn32, RPCEmu and frozen Debian kernels provide independent implementations that could not have copied this emulator's assumptions.

Failures are the useful result

If a specification-faithful device makes a guest fail, the failure is investigated on both sides. That distinction has already uncovered bugs in Linux, NetBSD and QEMU itself.

FINDINGS

What the emulated hardware exposed

LINUX

Floppy teardown ran work after releasing its resources

An FDC-less RISC PC reliably printed work still pending. The probe error path released IRQ and DMA state before draining its workqueue. Restoring the missing flush removes the race; the refined patch is ready for the block maintainers.

working Linux commit →
LINUX

The NetWinder could not claim its only disk controller

Footbridge described no PCI I/O window below 0x1000, so libata could never claim the SL82C105's fixed legacy IDE ports. Adding the DC21285's real 64 KiB window lets the machine mount its disk.

Linux patch →
LINUX

A configuration-dependent header build failure

With tracing enabled, linux/tick.h used ktime helpers and TICK_NSEC without including the headers that define them. The RISC PC config made the missing direct dependencies visible.

Linux patch →
QEMU

MMIO IDE silently latched an unconnected IRQ

QEMU boards normally connect sysbus IRQs after realizing a device. The MMIO IDE core copied the line too early, discarding every later interrupt. A forwarding IRQ restores conventional board wiring.

QEMU patch →
QEMU + NETBSD

An independent guest found two deeper faults

NetBSD exercised privileged SA-110 CP15 encodings Linux never reads, exposing an over-strict QEMU decoder. After that model fix it reached the clock and revealed an acorn32 divide-by-zero: stathz remains zero and is passed to the IOMD rate calculation.

evidence and instruction traces →
CURRENT

The working floppy found another legacy boundary

Real reads and writes now pass through the ARM FIQ pseudo-DMA path. They also show that mach-rpc's legacy enable_fiq() reaches generic IRQ startup with a descriptor that was installed but never activated. The transfer succeeds; the kernel warning remains recorded rather than hidden in the device model.

REPRODUCIBILITY

The demo is also the test case

The repository pins Linux and QEMU source commits, records every shipped binary's checksum and toolchain, and builds the page from plain static files. Before deployment, headless Chromium must boot the stable kernel, draw a non-empty 640×480 framebuffer, exercise physical and on-screen input, round-trip writes through IDE and floppy images, cut power, and restore the opt-in IndexedDB copy.
